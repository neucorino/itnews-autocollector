## [v1.4.0] - 2026-07-16

フロントエンド連携を見据えた「ユーザーパーソナライズ機能」の土台となるアップデートを行いました。ユーザーごとの関心トピックの保存や記事への「いいね」に対応するAPIエンドポイントを追加しました。あわせてデータアクセス層を整理し、データベース接続管理と整合性チェックを改善しています。

### Added
- パーソナライズ機能用のAPIエンドポイントを追加
  - `GET /v1/categories`: フロントエンド側でカテゴリー選択（チェックボックスなど）を表示するための、選択可能なIT分野一覧を返却。
  - `POST /v1/users/preferences`: ユーザーが選択した「興味のあるIT分野」を一括保存する（既存の設定を削除し、新規設定を再挿入する）エンドポイント。
  - `POST /v1/articles/{article_id}/like`: 記事に対する「いいね（お気に入り）」アクションを受け付けるエンドポイント。
- ユーザー管理・フィードバック用DBスキーマの追加
  - `users`(ユーザー情報) / `user_preferences`(ユーザーの関心トピック) / `article_feedbacks`(記事フィードバック) テーブルを新設。外部キー制約を設けることで、データの不整合を防止。
  - 設定やフィードバックを保存する際、データベース上に存在しないユーザーであれば自動でレコードを新規作成する仕組み（`ENSURE_USER`）を実装。
- データ操作（CRUD）ロジックの独立
  - `crud.py` を新設し、記事の分析・ランキング・通知対象・ユーザー設定・フィードバックなどのデータ操作を集約。
- データ検証用モデルの追加 
  - `UserPreference` / `ArticleFeedback`（内部のデータ構造を定義するdataclass）
  - `UserPreferencesRequest` / `ArticleLikeRequest`（APIからのリクエストを検証するPydanticモデル）

### Changed
- データベース接続のライフサイクル管理と堅牢化
  - `DatabaseManager` の役割を「接続管理・スキーマ定義・バッチ処理」に限定し、データ操作（CRUD）は新設の `crud.py` に委譲。これに伴い `service.py` や `api.py` も追随。
  - 処理が正常終了した場合は自動コミット、エラー時はロールバックする、安全な接続管理の仕組み（`get_connection` コンテキストマネージャ）を導入。
  - FastAPI側のDBセッション（`get_db`）もコンテキストマネージャ化し、リクエスト処理終了時に接続が確実にクローズされるよう変更。
  - SQLiteで外部キー制約（`PRAGMA foreign_keys = ON`）を有効化し、テーブル間のデータ参照の整合性を保証。
- 既存API（`GET /news`）の仕様スリム化と初期UXの最適化
  - リクエストパラメータから、将来的に使用しないページング用の `skip` パラメータを削除（API仕様のデッドコードを排除）。
  - ユーザーがパラメータを指定せずとも、初回アクセス時に「重要度7以上の厳選された最新ニュース10件」が即座に返却されるよう、デフォルト値を `limit=10`、`min_importance=7` へ最適化。


## [v1.3.2] - 2026-07-10

複数 RSS ソースへの対応と、ユーザー関心トピックに基づくパーソナライズスコアリングを導入。日付パースの堅牢化、Gemini リトライの強化、記事 ID 重複の防止により、マルチソース運用時の安定性を向上しました。

### Added
- RSS ソースの追加: TechCrunch / The Verge / Ars Technica / GitHub Blog（既存の Hacker News に加え、計 5 ソースから収集）
- `USER_PREFERENCES` の導入: ユーザーの関心トピックをプロンプトに反映し、パーソナライズされた重要度スコアリングに対応
- `parse_rss_date` の追加: RFC 822 / ISO 8601 の両形式に対応し、パース失敗時は現在時刻へフォールバック
- `article_analyses` テーブルに `(article_id, batch_id)` の UNIQUE 制約およびユニークインデックスを追加

### Changed
- ソース別取得上限を `SOURCE_FETCH_LIMIT`（20 件）として分離し、全ソースの記事を集約してから 1 回だけ分析・ランキングを実行するフローへ変更
- Gemini API のリトライ強化: 503 / UNAVAILABLE もリトライ対象に追加、指数バックオフを採用、失敗時は重要度 0 のダミーレコードを生成してバッチ全体を継続

### Fixed
- `INSERT OR IGNORE` でスキップされた際に誤った `article_id` が割り当てられる不具合を修正（`rowcount` 判定と URL からの ID 再取得）
- Gemini 分析前に `article_id` で重複排除し、同一記事の二重分析を防止


## [v1.3.1] - 2026-06-24

Docker Compose による API サーバーの死活監視と自動復旧を強化。ヘルスチェックエンドポイントの追加と Compose 側の運用設定を整備し、コンテナ環境でのサービス信頼性を向上しました。

### Added
- `GET /health` エンドポイントの追加: Docker Engine 向けの API サーバー死活監視用（`{"status": "healthy"}` を返却）
- オンデマンド実行用にコンポーネント分離した `worker` コンテナを、cronによる自律定期実行を設定。
- ホスト側の専用ログファイル（`cron.log`）へ、コンテナの標準出力およびエラー出力（`2>&1`）をすべて集約

### Changed
- `Dockerfile` に `curl` を追加: Compose `healthcheck` コマンドの実行に必要
- `compose.yaml` の `api` サービスに `healthcheck` を設定: 30 秒間隔で `/health` を確認し、3 回連続失敗で unhealthy と判定
- `compose.yaml` の `api` サービスに `restart: always` を追加: プロセス異常終了や unhealthy 判定時の自動再起動

## [v1.3.0] - 2026-06-22

Docker (Compose v2) によるコンテナ環境への完全対応。アプリケーションコードへの変更を最小限に抑えつつ、ビルドから API サーバー起動までを 1 コマンドで完結できる実行基盤を構築しました。

### Added
- `Dockerfile` の追加: `python:3.12-slim` ベース、`PYTHONPATH=/app` による絶対インポートの固定、レイヤーキャッシュ最適化（`requirements.txt` 先行 COPY）
- `compose.yaml` の追加: `api`（FastAPI 常時起動）と `worker`（収集バッチ・オンデマンド実行）のサービス分離
- ホストボリュームマウント: `data/` / `logs/` をコンテナと共有し、バッチ処理結果を API サーバーへ即時反映
- `.env` による秘密情報のコンテナ注入: `env_file` 経由で API キー・SMTP 認証情報を読み込み、イメージへの埋め込みを回避
- `worker` サービスの `profiles: ["manual"]` 設定: `docker compose up -d` 時の自動起動を抑制し、必要時のみバッチ実行

### Changed
- セットアップ手順を Docker Compose v2 前提に刷新: `docker compose up -d` による API 起動、`docker compose run --rm worker` によるバッチ実行
- README に Dockerfile 設計解説・Docker 関連トラブルシューティングを追加

## [v1.2.1] - 2026-06-11

### Added
- 段階的鮮度係数モデル（`FRESHNESS_TABLE`）の導入（公開日時からの経過日数に応じてスコアが段階的に低下）
- `rank_score` の算出・DB保存（重要度スコア × 鮮度係数）
- 同一スコア時の安定ソート（優先順位：公開日時 → ID）
- メール本文への記事公開日時の追加

### Changed
- 公開日時の保存形式を RFC 2822 から `YYYY-MM-DD HH:MM:SS`（ISO 8601）へ正規化
- Gemini API モデルを Gemini 2.5 Flash から Gemini 2.5 Flash-Lite へ変更（API利用料金の削減）
- Temperature を引き上げ、重要度スコア判定の柔軟性を向上
- プロンプトをモデル変更にあわせて見直し


## [v1.2.0] - 2026-05-27
既存のバッチ処理ロジックの資産を完全に活かしつつ、REST API 機能を独立したコンポーネントとしてシームレスに統合。疎結合なアーキテクチャへの拡張を行いました。

### Added
- FastAPIによるREST API化: `GET /v1/rankings` および `GET /v1/notifications` エンドポイントの実装。
- CORS（Cross-Origin Resource Sharing）設定: `CORSMiddleware` を導入し、クロスオリジン接続の安全な制御に対応。
- `__init__.py` の配置: ディレクトリ構造を明示的にパッケージ化し、モジュール探索を最適化。

### Changed
- Pydantic v2 レスポンスモデルの統合: 各APIレスポンスの型安全性を保証。
- 絶対インポートへの刷新: 実行環境に依存しない堅牢なモジュール解決のために、プロジェクトルート基準のインポートパスへ全面修正。
- 通知ロジックとAPIレスポンスのブラッシュアップ: 仕様変更に伴う調整。


## [v1.1.1] - 2026-04-24
堅牢性の向上とリファクタリングを中心としたメンテナンスリリース。

### Added
- 設定値の妥当性検証機能 (`validate_config`): 起動時に設定エラーを即座に検知するフェイルファスト設計を導入。
- Gemini API 呼び出し時の自動リトライ処理: 一時的な接続エラー（テンポラリエラー）への耐性を向上。
- モジュール分離の強化:  `constants.py` / `exceptions.py` / `queries.py` を新設し、凝集度と可読性を向上。

### Changed
- `DatabaseManager` の強化: コンテキストマネージャー（`with` 構文）に対応し、確実なリソース解放を保証。
- エラーハンドリングの適正化: バッチ全体の例外捕捉ロジックを整理し、`logger.py` から重複ハンドラを削除。


## [v1.1.0] - 2026-04-17
バッチ処理ロジックの「関心の分離」を目的とした、大幅なアーキテクチャ刷新。

### Added
- `NewsService` クラス（Service層）の導入: ビジネスロジック（データ保存、通知対象抽出、フロー制御）を`service.py`に集約し、責務を分離。
- 依存注入（DI）パターンの採用: `DatabaseManager` をコンストラクタ引数で注入する設計へ変更し、単体テスト時のモック化を容易に。
- `Ranking`処理の分離: ランキング生成ロジックを独立したモジュールへ分離。

### Changed
- メール送信処理の最適化: メール本文の自動テンプレート化を施し、送信エラー時のリトライ機能を実装。
- APIモデルのアップデート: GeminiモデルのIDを `gemini-2.5-flash` へ刷新し、処理性能を最適化。
- 記事取得処理の制限: API消費を抑えるため、1回あたりの取得上限を設定。

コア機能（ニュース収集・分析・ランキング生成・通知）が揃ったファーストメジャーリリース。

### Added
